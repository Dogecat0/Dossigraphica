# **Architecture and Implementation of a Local Deep Research Agent Pipeline in TypeScript**

**Anchor Date:** April 4, 2026\.

The following intelligence briefing and architectural blueprint executes an exhaustive forensic investigation into the deployment of a fully localized, zero-cost "Deep Research" pipeline integrated into a TypeScript-based application. The structural analysis relies exclusively on primary deployment data, local large language model (LLM) hardware benchmarking, state-of-the-art agentic orchestration frameworks, and advanced open-source search infrastructure parameters available as of the second quarter of 2026\.

The assessment is designed to provide principal software engineers and AI systems architects with a granular, hardware-aware understanding of the specific operational frameworks required to run autonomous reasoning agents natively on consumer-grade silicon, specifically the NVIDIA RTX 3070 with 8GB of dedicated Video RAM (VRAM) and 24GB of system memory.1

## **MODULE A: Hardware Constraints and Sovereign Silicon Optimization**

The physical and mathematical limitations of consumer-grade silicon dictate the entire architectural approach to localized AI agents. Operating a multi-hop, highly iterative deep research pipeline within a strict 8GB VRAM ceiling requires aggressive quantization, precise context window management, and strict avoidance of shared memory fallback. The RTX 3070 features 8192 Megabytes of total VRAM, of which approximately 500 to 800 Megabytes are permanently reserved by the host operating system for display overhead and background rendering tasks.1 This leaves a functional computing envelope of approximately 7.2GB to 7.4GB for the LLM inference engine.

When a localized LLM exceeds this dedicated VRAM threshold, modern inference frameworks such as Ollama or llama.cpp seamlessly spill the active model layers and the Key-Value (KV) cache into the system's 24GB of shared RAM.3 While this architectural fallback prevents the application from crashing with a fatal Out-of-Memory (OOM) error, it introduces severe PCIe bus latency.3 The system RAM operates at a fraction of the bandwidth of dedicated GDDR6 VRAM. Consequently, throughput routinely plummets from highly interactive, real-time generation speeds of 40 to 50 tokens per second (t/s) down to an unworkable 1 to 5 t/s.3 For an autonomous deep research agent that must iterate through tens of thousands of tokens across planning, searching, reading, and synthesizing phases, relying on system RAM offloading effectively paralyzes the pipeline. Therefore, the total memory footprint—comprising the quantized model weights, the inference engine overhead, and the KV cache required for the context window—must remain strictly under the 7.4GB threshold to maintain operational viability.1

To manage this strict boundary, systems engineers must manipulate the parameters of the local inference engine. Within the Ollama ecosystem, the num\_gpu parameter dictates the maximum number of neural network layers offloaded directly to the physical GPU.1 While maximizing this number ensures peak token generation speed, over-allocating layers on an 8GB card when processing massive context windows will force the KV cache into system RAM.6 Furthermore, the OLLAMA\_NUM\_PARALLEL environment variable must be tightly controlled.7 Deep research pipelines frequently execute parallel asynchronous requests, but permitting the LLM to process multiple simultaneous streams multiplies the KV cache footprint linearly.8 On an 8GB RTX 3070, executing more than one concurrent generation stream with a context window exceeding 8,000 tokens guarantees a VRAM overflow.5 The optimal configuration forces the TypeScript application to handle concurrency at the network level (e.g., parallel web scraping) while strictly queueing LLM inference tasks sequentially.

## **MODULE B: State-of-the-Art Local Models for 8GB VRAM Inference**

To achieve deep, autonomous reasoning capabilities within an 8GB envelope, the deployment must leverage models residing in the 3-billion to 9-billion parameter class, specifically utilizing 4-bit quantization algorithms such as GGUF Q4\_K\_M.5 As of April 2026, the open-source ecosystem has achieved unprecedented "intelligence density" through advanced model distillation and parameter-efficient architectures, allowing consumer hardware to rival the performance of enterprise-grade cloud APIs.10

The first viable candidate for the localized deep research agent is the DeepSeek-R1-Distill-Qwen-7B model. Distilled directly from the massive 671B DeepSeek-R1 reinforcement learning model onto a highly efficient Qwen 2.5 base architecture, this 7-billion parameter variant requires approximately 4.3GB of VRAM at Q4 quantization.12 This lightweight footprint leaves a comfortable buffer of approximately 3.1GB specifically for the KV cache, enabling context windows of up to 24,000 tokens before performance degradation occurs via system RAM offloading.5 The primary advantage of the DeepSeek-R1 lineage is its native reasoning capability; the model is explicitly trained to output its internal logical deductions prior to generating a final answer, allowing the agent to "think" before acting.11 This makes it an exceptional candidate for the planning and reflection stages of the research loop.

The second, and arguably most optimized candidate for this specific hardware profile, is the Google DeepMind Gemma 4 (E4B) model, officially released on April 2, 2026\.16 Released under the commercially permissive Apache 2.0 license, the E4B (Effective 4 Billion parameters) variant is purpose-built for on-device agentic workflows.16 Utilizing a novel architectural mechanism known as Per-Layer Embeddings (PLE), the model requires only 5.5 to 6GB of VRAM at 4-bit quantization while supporting a massive native context window of 128K tokens.19 Furthermore, Gemma 4 introduces native function-calling, interleaved multimodal input capabilities, and strict adherence to structured JSON output generation, which is mathematically advantageous for programmatic integration within a TypeScript environment.16 It is critical to note that while Google released a highly performant Gemma 4 26B-A4B Mixture-of-Experts (MoE) model that only activates 3.8B parameters during inference, the entirety of the 26B parameter matrix must still be loaded into memory, demanding roughly 16GB of VRAM at Q4 quantization, completely disqualifying it from the target 8GB hardware profile.17

The third candidate is the Qwen 3.5 9B dense model. Released in early 2026, this model pushes the absolute mathematical limit of the 8GB RTX 3070, consuming roughly 6.49GB of VRAM at standard Q4 quantization.9 While the memory margin is razor-thin, Qwen 3.5 utilizes a Gated DeltaNet hybrid attention architecture that alternates linear and full attention in a 3:1 ratio, vastly reducing KV cache bloat.23 This allows the model to process large blocks of scraped web text more efficiently than standard dense transformer architectures, making it highly suitable for the data extraction and summarization phases of the deep research pipeline.23

## **MODULE C: The Sovereign Web Search and Extraction Infrastructure**

A deep research pipeline is fundamentally bottlenecked by its ability to reliably access, extract, and format real-time internet data without triggering rate limits, bot-protection captchas, or recurring financial costs.25 Relying on proprietary, managed APIs such as Tavily, Perplexity Sonar, or Exa introduces direct per-query costs, violating the foundational requirement of a zero-cost architecture.27 Even utilizing the hosted API of open-source platforms like Firecrawl incurs credit exhaustion after the initial free tier.27 To satisfy the mandate of a 100% free, localized system, the search infrastructure must be completely decoupled from commercial Software-as-a-Service (SaaS) dependencies and run entirely within the host machine's execution environment.

The core of this autonomous retrieval system is SearXNG, a robust, open-source, privacy-preserving metasearch engine that aggregates results from over 70 distinct search providers, including Google, Bing, Brave, and academic repositories like arXiv and PubMed.29 Crucially, SearXNG operates without tracking user IP addresses or requiring external API keys, making it immune to vendor lock-in or sudden pricing shifts.29 For an autonomous agent operating within a TypeScript application, SearXNG serves as the ultimate programmatic gateway to the live internet.

To securely integrate SearXNG with the TypeScript backend, it must be deployed locally via Docker and explicitly configured to output machine-readable data structures rather than visual web components. By default, SearXNG returns HTML web pages designed exclusively for human browsers.31 To enable the JSON API essential for the TypeScript application's parsing logic, the instance's settings.yml file must be manually modified to expose the json format parameter within the search.formats array.31 The infrastructure requires a localized Docker Compose stack isolating the SearXNG container on the loopback interface (localhost / 127.0.0.1) to prevent external network exposure and maintain strict operational security.29 The search step within the TypeScript application then executes a native HTTP GET request utilizing the format http://localhost:8080/search?q={url\_encoded\_query}\&format=json.32 The resulting JSON payload provides the deep research agent with an array of URLs, page titles, and textual snippets, serving as the navigational blueprint for the subsequent scraping phase.

Once the agent identifies high-value URLs via the SearXNG payload, it must extract the raw HTML and convert it into a token-efficient format, specifically clean Markdown. The industry standard for robust, LLM-ready web extraction is Firecrawl, an open-source web crawler designed specifically for AI agents.28 Firecrawl natively handles JavaScript-rendered content, circumvents anti-bot protections, and outputs perfectly formatted Markdown.35 However, deploying Firecrawl as a self-hosted solution introduces a massive infrastructure footprint, demanding concurrent Redis clusters, PostgreSQL databases, and resource-intensive Playwright browser worker containers.37 This severe architectural overhead competes directly with the localized LLM for the host machine's limited CPU cycles and system RAM, risking overall system instability and catastrophic inference latency.37

For a TypeScript application focused purely on frontend mechanics with minimal backend complexity, the optimal zero-cost, low-resource extraction architecture involves utilizing lightweight libraries directly within the Node.js or Next.js runtime environment, entirely bypassing heavy Dockerized browser automation.39 The pipeline executes a standard asynchronous fetch() request to retrieve the target HTML, routes the raw markup through a virtual DOM environment like JSDOM, and applies Mozilla's @mozilla/readability library.40 This library algorithmically identifies and isolates the primary article content, mathematically stripping away navigation bars, advertisements, footer boilerplate, and cookie banners. Finally, the TurndownService library converts the sanitized HTML fragments into clean Markdown.41 While this lightweight approach fails on heavily obfuscated Client-Side Rendered (CSR) single-page applications that require headless browser execution, it boasts a near 100% success rate on standard news sites, corporate press releases, SEC filings, and Wikipedia articles. Crucially, this pipeline requires zero additional Docker containers, incurs zero API costs, and executes in milliseconds, preserving the host machine's compute budget entirely for the LLM inference engine.

## **MODULE D: Orchestration Architecture and The Deep Research State Machine**

Deep research transcends simple, linear prompt-and-response paradigms. It is a highly cyclical, asynchronous, and stateful process that mimics human forensic investigative behavior: formulating an initial hypothesis, searching the web, reading diverse documents, recognizing that the retrieved information is incomplete, pivoting the search query based on new context, and continuously refining the underlying knowledge base until a definitive, mathematically sound conclusion is reached.26

To effectively manage this complex multi-agent reasoning loop within a TypeScript application without relying on heavy, dependency-laden frameworks, the architecture must rely on state machine orchestration, specifically leveraging the LangGraph.js framework.43 Unlike older, monolithic chain-based frameworks that execute rigid, linear sequences, LangGraph models the application as a Directed Acyclic Graph (DAG).44 It treats each distinct step of the research process as an isolated, functional "node" that consumes, modifies, and returns a shared "state" object, allowing for conditional routing and cyclic loops.44

The entire intelligence and memory of the application live within a persistent, strictly typed data structure. In TypeScript, this is defined via an interface, establishing the precise boundaries of the agent's working memory. The ResearchState object must be engineered to hold all historical and active data points:

* user\_query: The original prompt provided by the human user.  
* research\_plan: A string detailing the strategic methodology generated by the LLM.  
* search\_queries: An array of highly specific strings to be executed against the SearXNG endpoint.  
* raw\_extracted\_data: A continuously appending array of Markdown text blocks fetched from target URLs, including strict citation metadata.  
* iteration\_count: An integer tracking the number of cyclical search loops to prevent infinite execution loops and unbounded compute consumption.  
* final\_report: The ultimate synthesized Markdown output.

The local TypeScript application routes this state object through a meticulously choreographed sequence of six specialized nodes, each handling a unique phase of the investigation 38:

1. **The Strategy Planner Node:** The local reasoning LLM (e.g., DeepSeek-R1-Distill or Gemma 4\) ingests the raw user\_query. Instead of attempting to answer the question directly, it acts as a research director, generating a tactical research\_plan and outputting a structured JSON array containing three to five highly specific search\_queries designed to cover disparate facets of the geopolitical or financial topic.48  
2. **The Parallel Search Node:** The TypeScript application executes rapid, parallel HTTP GET requests to the local SearXNG Docker container for each query residing in the search\_queries array. It parses the JSON response and extracts the top three to five highly relevant URLs per query, discarding duplicate domains.44  
3. **The Fetch & Scrape Node:** The application concurrently fetches the raw HTML from the identified URLs. It utilizes the aforementioned mozilla/readability and Turndown protocol to strip algorithmic noise and convert the raw HTML structures into clean, token-efficient Markdown.40  
4. **The Synthesizer Node:** This node addresses the severe hardware limitations of the RTX 3070\. Because the agent cannot dump ten full Markdown articles into the LLM's context window without overflowing the 8GB VRAM capacity, the system executes a mathematical compression phase. The application passes individual Markdown chunks to the LLM, prompting it to extract only the facts strictly relevant to the user\_query, discarding all tangential information. These dense micro-summaries are then appended to the raw\_extracted\_data array within the state object.  
5. **The Reflector Node:** The LLM evaluates the newly updated raw\_extracted\_data against the original user\_query. It acts as an adversarial judge, rigorously determining if the current knowledge base contains factual gaps, logical contradictions, or missing geographical citations.49  
6. **The Routing Edge (Conditional Logic):** A conditional logic gate embedded within the LangGraph framework evaluates the Reflector Node's output. If critical knowledge gaps remain and the iteration\_count is below the hardcoded threshold (e.g., maximum 3 loops), the graph forcefully routes the state back to the Planner Node to generate new, highly targeted search queries designed specifically to fill the identified voids.50 If the data is deemed sufficient, the graph breaks the loop and routes to the Final Publisher Node to draft the exhaustive 5000-word comprehensive report based entirely on the curated facts.50

## **MODULE E: Context Window Defense and Recursive Summarization**

The most critical vulnerability of any localized deep research architecture is catastrophic context bloat.51 A single scraped web page, even when perfectly stripped of HTML boilerplate and converted to Markdown, can easily contain between 3,000 and 5,000 tokens of text. If an autonomous agent loops three times, scraping five unique URLs per loop, the raw text volume rapidly scales to an overwhelming 45,000 to 75,000 tokens. While advanced models like Gemma 4 technically support massive native context windows of up to 128K tokens 52, processing a 60,000-token prompt requires an enormous amount of physical VRAM dedicated purely to maintaining the Key-Value (KV) cache. Loading a 60,000-token context onto an 8GB RTX 3070 guarantees a memory overflow, instantly forcing the inference engine to utilize system RAM and destroying generation latency.5

To mathematically defend the hardware constraints, the TypeScript application must implement strict data compression algorithms, formally known as "Recursive Summarization" or "Context Squeezing," primarily executed within the Synthesizer Node.54 Raw HTML or unabridged Markdown from scraped websites is never passed directly to the final report generator. Instead, as each webpage is scraped, a highly constrained, ephemeral instance of the local LLM evaluates the text against the core objectives of the user\_query. The prompt explicitly instructs the model to extract only the hard facts, statistics, geographical locations, and actionable intelligence, aggressively discarding narratives, redundant explanations, and tangential data.55

Crucially, during this extraction phase, the system programmatically appends the original source URL citation directly to the extracted facts to maintain forensic data provenance.54 A sprawling 4,000-token web article is thereby mathematically compressed into a dense, high-signal 200-token bulleted summary. By the end of a sprawling, multi-iteration research loop, the final context window presented to the Publisher Node contains only the highly concentrated, perfectly cited summaries of perhaps 15 to 20 distinct web pages, safely fitting within a highly manageable 4,000 to 6,000-token footprint. This meticulous memory management ensures the final VRAM load remains stable, preventing PCIe bus latency and allowing the localized 8GB agent to output a flawless, comprehensive intelligence briefing at maximum inference speed.51

## **MODULE F: Enforcing Deterministic JSON Outputs in TypeScript**

The fundamental point of failure for locally hosted, sub-10 billion parameter LLMs operating within an autonomous agentic pipeline is structural hallucination. The LangGraph state machine relies on programmatic interoperability; when the TypeScript application expects an array of search queries to pass to the SearXNG endpoint, but the LLM returns a conversational preamble (e.g., "Certainly\! Here are the search queries you requested based on your prompt:"), the application's parsing logic will immediately suffer a fatal crash.39 The inference models must be strictly and mercilessly coerced into outputting valid, parseable JSON payloads without exception.

As of the second quarter of 2026, the most resilient architectural mechanism for enforcing structured data generation in a local TypeScript environment combines the ollama-js SDK with the Zod validation library.56 Rather than relying on fragile, complex Regex string manipulation algorithms to extract JSON blocks from conversational text, the application leverages the zod-to-json-schema utility to mathematically dictate the exact output shape the LLM is permitted to generate at the token-generation level.58

Within the TypeScript architecture, every node in the LangGraph that requires structured programmatic data (such as the Planner Node generating queries, or the Reflector Node outputting boolean flags) must define a strict Zod schema. For example, the Planner Node defines its expected output utilizing standard TypeScript type definitions 56:

TypeScript

import { z } from 'zod';  
import { zodToJsonSchema } from 'zod-to-json-schema';

const PlannerSchema \= z.object({  
  analysis: z.string().describe('Internal analytical reasoning regarding the user query.'),  
  search\_queries: z.array(z.string()).describe('An array of exact search strings to query the internet.'),  
});

The application then passes this schema definition directly into the Ollama chat request payload utilizing the format parameter:

TypeScript

const response \= await ollama.chat({  
  model: 'gemma-4-e4b',  
  messages:,  
  format: zodToJsonSchema(PlannerSchema),  
  options: { temperature: 0.1 }  
});

This API structure forces the local Gemma 4 or Qwen 3.5 model to constrain its token selection probabilities exclusively to valid JSON characters that strictly adhere to the defined schema keys, completely eliminating conversational preambles and ensuring flawless execution within the Node.js runtime.57

A unique architectural anomaly arises, however, when utilizing DeepSeek-R1 distilled models for JSON generation tasks. Because these reasoning models are explicitly trained via large-scale reinforcement learning to output their internal "Chain of Thought" (CoT) logic prior to generating their final answer, they natively inject \<think\> and \</think\> tags directly into the response stream.15 If the Ollama SDK attempts to execute standard JSON.parse() on a string payload that begins with \<think\> evaluating search parameters for optimal retrieval... \</think\> { "search\_queries": \[...\] }, the JavaScript runtime will throw a fatal syntax exception.61 To securely utilize the vastly superior reasoning capabilities of the DeepSeek-R1 lineage within the automated pipeline, the TypeScript application must implement a custom interceptor function. This function intercepts the raw string response, algorithmically locates the closing \</think\> tag, aggressively discards all preceding textual data, and parses only the remaining string buffer as the verified JSON payload.61

## **MODULE G: Advanced Prompt Engineering for Local Reasoning Models**

The deployment of models such as DeepSeek-R1-Distill-Qwen-7B and Gemma 4 (E4B) fundamentally alters the paradigm of prompt engineering within agentic systems. Historically, developers utilizing older models were forced to inject extensive, verbose instructions into system prompts to coerce logical deduction, utilizing phrases such as "Think step-by-step," "Break this problem down into smaller components," or "Analyze the data carefully before responding".62

Reasoning-first models possess these analytical behaviors intrinsically, having been baked into their weights during post-training reinforcement learning.11 Injecting traditional, heavy-handed "step-by-step" prompt structures into these models often severely degrades their performance, causing them to enter endless repetition loops, hallucinate formatting, or become confused by conflicting internal reinforcement pathways.11 For the local deep research pipeline, prompt architecture must shift from dictating *how* the model should think to establishing highly declarative boundaries regarding constraints, required output formatting, and strict source attribution.63

The system prompts must operate as rigid legal frameworks. For instance, the prompt architecture governing the critical "Reflector Node" must be constructed as follows:

**Role Authorization:** You are an expert fact-checker and forensic research validator operating within an automated geopolitical intelligence pipeline. **Operational Task:** Analyze the provided raw\_extracted\_data array against the initial user\_query. **Constraint 1 (Reasoning Channel):** You must output your internal evaluation using the designated reasoning tags prior to your final decision. For DeepSeek-based models, strictly utilize the \<think\> and \</think\> tags. For Gemma 4 architectures, utilize the \<|channel\>thought\\n and \<channel|\> tags.17 **Constraint 2 (Gap Identification):** Rigorously identify specific factual gaps in the retrieved data. Are there corporate claims lacking statistical evidence? Are critical dates missing? Are facility locations unverified? **Constraint 3 (Routing Flags):** If the retrieved data completely satisfies all facets of the user query, output is\_complete: true within your JSON payload. If factual gaps exist, output is\_complete: false and provide exactly three new, highly targeted search\_queries designed to fill the identified voids. **Constraint 4 (The Citation Rule):** Every factual claim must be mathematically tied to its source URL.

This highly constrained, boundary-focused prompt architecture ensures that the local 8GB model focuses its limited VRAM compute budget entirely on evaluating data quality and determining execution routing, rather than wasting memory processing overly verbose, unnecessary system instructions.63 By strictly enforcing the execution parameters, optimizing the search retrieval methodology, and rigorously defending the KV cache limits, a standard consumer workstation can reliably execute deeply autonomous, forensic-level research that rivals enterprise cloud solutions, operating at zero recurring cost and maintaining absolute data sovereignty.
